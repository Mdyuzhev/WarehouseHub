== Цели и Scope тестирования

=== Цели тестирования

==== Основная цель

[IMPORTANT]
====
Определить максимальную пропускную способность Warehouse API и количественно измерить влияние Redis кэширования и Kafka на производительность системы.
====

==== Конкретные цели

. *Найти точку насыщения* — максимальное количество concurrent users до деградации
. *Измерить пропускную способность* — Requests Per Second (RPS) при различных нагрузках
. *Оценить латентность* — время отклика (p50, p95, p99) на каждом уровне нагрузки
. *Сравнить режимы* — количественная разница между Baseline и Optimized
. *Выявить узкие места* — определить bottlenecks в системе

=== Критерии успеха

==== SLA (Service Level Agreement)

[cols="2,2,2,3", options="header"]
|===
| Метрика | Целевое значение | Критическое | Действие

| Error Rate
| < 1%
| > 3%
| STOP тест, анализ ошибок

| Response Time p95
| < 500ms
| > 1500ms
| STOP тест, профилирование

| Response Time p99
| < 1000ms
| > 3000ms
| Warning, мониторинг

| RPS
| > 100 req/s
| < 50 req/s
| Анализ bottlenecks
|===

==== Критерии завершения теста

[source]
----
АВТОМАТИЧЕСКАЯ ОСТАНОВКА:
├── Error Rate > 3% в течение 2+ минут
├── p95 Response Time > 1500ms в течение 2+ минут
├── Connection Refused (система не отвечает)
└── OOM Kill (контейнер убит из-за памяти)

ПЛАНОВАЯ ОСТАНОВКА:
├── Достигнут максимум профиля (2000 users)
├── Время теста > 80 минут
└── Все метрики собраны на всех уровнях нагрузки
----

=== Риски и митигация

[cols="2,1,3,3", options="header"]
|===
| Риск | Вероятность | Влияние | Митигация

| Тест на Production
| Высокая
| Критическое
| Проверять HOST перед запуском, hardcoded проверка в скрипте

| Переполнение БД тестовыми данными
| Средняя
| Среднее
| Cleanup в on_stop() каждого user, DELETE созданных продуктов

| Высокая
| Ожидаемое

| Network throttling
| Низкая
| Среднее

| Redis/Kafka недоступны
| Низкая
| Высокое
| Health check перед тестом, fallback в логах

== Профили нагрузки

[source]
----

├── 5-10 мин:  25 users   (spawn_rate=5)    — Лёгкая нагрузка
├── 10-15 мин: 50 users   (spawn_rate=10)   — Умеренная нагрузка

├── 20-25 мин: 150 users  (spawn_rate=20)
├── 25-30 мин: 200 users  (spawn_rate=20)   — Повышенная нагрузка
├── 30-35 мин: 250 users  (spawn_rate=20)
├── 35-40 мин: 300 users  (spawn_rate=25)

=== StabilityLoadShape (проверка стабильности)

==== Описание

[source]
----
Цель: Проверить стабильность системы при 80% от найденного максимума

ПАРАМЕТРЫ:
├── Target Users: 80% от max_users (найденного в StepLoad)
├── Ramp-up: 10 минут
├── Steady State: 60 минут
├── Ramp-down: 5 минут
└── Total Duration: 75 минут

ПРИМЕР (если max_users = 300):
├── 0-10 мин:  Ramp-up до 240 users
├── 10-70 мин: Steady state 240 users
└── 70-75 мин: Ramp-down до 0
----

==== Критерии стабильности

[cols="2,2,3", options="header"]
|===
| Метрика | Порог | Условие успеха

| Error Rate
| < 1%
| Стабильно на протяжении 60 минут

| p95 Latency
| < 500ms
| Без деградации во времени

| RPS Variance
| < 10%
| Стабильная пропускная способность

| Memory
| Без утечек
| Heap использование стабильно

==== Обоснование

[source]
----
Распределение отражает реальное использование системы:

=== Выявленный Bottleneck: PostgreSQL Connections

[WARNING]
====
**4 реплики API × 20 connections = 80 соединений**

При PostgreSQL max_connections=100 система достигает ~80% лимита.
При высокой нагрузке HikariCP ждёт освобождения connections → latency растёт.
====

==== Диагностика

[source,bash]
----
# PostgreSQL max connections
kubectl exec -n warehouse postgres-0 -- psql -U postgres -c "SHOW max_connections;"
# Результат: 100

# Текущие соединения (при 300 users)
kubectl exec -n warehouse postgres-0 -- psql -U postgres -d warehouse -t \
  -c "SELECT count(*) FROM pg_stat_activity WHERE datname='warehouse';"
# Результат: 81

# HikariCP настройки
SPRING_DATASOURCE_HIKARI_MAXIMUM_POOL_SIZE=20  # × 4 pods = 80 max
----

==== Анализ

[source]
----
Почему 4 реплики хуже 2 реплик:

┌─────────────────────────────────────────────────────────────────┐
│  Test C (2 replicas)           Test D (4 replicas)              │
│  ────────────────────          ────────────────────             │
│  2 pods × 10 conn = 20 max     4 pods × 20 conn = 80 max        │
│  PostgreSQL: 20/100 = 20%      PostgreSQL: 80/100 = 80%         │
│                                                                 │
│  При 150 users:                При 150 users:                   │
│  - Connections: ~15            - Connections: ~60               │
│  - Нет contention              - Contention начинается          │
│                                                                 │
│  При 300 users:                При 300 users:                   │
│  - Connections: ~25            - Connections: ~81 (лимит!)      │
│  - Ещё есть запас              - HikariCP ждёт → latency ↑      │
└─────────────────────────────────────────────────────────────────┘

Вывод: Больше реплик ≠ лучше, если bottleneck смещается на БД.
----
